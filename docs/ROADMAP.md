# Bounce ÔÇö Production MVP Roadmap

Path from the current working demo to a production-ready, **multi-tenant** SaaS that can take real customers and pay real operators.

_Last updated: 2026-07-15._

## Where we are

The core product loop works end-to-end and is deployed (https://bounce-app.vercel.app):
conversational AI quote Ôćĺ deposit/full checkout (Stripe) Ôćĺ e-signed booking (SignWell)
Ôćĺ operator dashboard / calendar / inbox, all on live data.

**Solid today**
- Booking + availability engine (date ranges, peak-reservation math, oversell detection, abandoned-checkout expiry).
- Stripe Checkout with idempotent webhooks; SignWell agreement send path built + verified (currently **off in prod** ÔÇö test mode; see Go-Live Checklist).
- Conversational AI quote agent grounded in live inventory + DB prices.
- Operator app + storefront, wired to real data, responsive, deployed.

**Known gaps (verified in code)**
- **No RLS policies** ÔÇö RLS is *enabled* on the tenant tables but the only policy in the whole migration set is `operator_members` "members read own" (`0008`). Every read/write goes through `createAdminClient()` (service role, bypasses RLS), so **tenant isolation lives entirely in application code**: one missed `.eq("operator_id", ÔÇŽ)` leaks another operator's data with no second line of defense. Tracked under Multi-tenancy (Tier 0).
- **Migrations are hand-applied** ÔÇö CLI-tracked (`./scripts/db-migrate.sh`), but not part of the deploy.
- ~~No authentication~~ Ôťů Supabase Auth + `operator_members` + middleware gating. ~~Single Stripe account, no Connect~~ Ôťů Express connected accounts + destination charges. ~~No transactional email~~ Ôťů Resend. ~~In-memory rate limiting~~ Ôťů Upstash Redis (sliding window). ~~No automated tests / CI~~ Ôťů **Vitest (207 tests) + GitHub Actions CI**. ~~Shared devÔćöprod DB~~ Ôťů split into separate projects.

### The one code item that isn't on this list ÔÇö and should worry you

**RLS.** Tenant isolation lives *entirely* in hand-written application code: every query goes through the service-role client, so one forgotten `.eq("operator_id", ÔÇŽ)` leaks another operator's data with no second line of defense. **This pattern has now bitten twice** ÔÇö both times harmlessly, both times by luck rather than design:

1. The middleware gate was missing `/customers`, `/promos`, `/documents`, `/account` (fixed `5afc478`; nothing leaked only because each page happened to guard itself ÔÇö now enforced by a test that reads the route group off disk).
2. Every renter-portal query had to be hand-scoped by account, and correctness was established by writing tests for each one rather than by the database refusing to serve the wrong rows.

It won't always be luck. Before real operators with real customer lists, this deserves the belt as well as the braces. _(Tracked under Multi-tenancy, Tier 0.)_

- **DECISION ÔÇö keep the marketing site inside the app; split at DNS, not at the codebase** _(2026-07-14)_

  **Where it lives today:** a `(marketing)` route group in the same Next.js app, **statically prerendered** (`ÔŚő` in the build output) and served from Vercel's CDN. The middleware skips those paths entirely, so they cost nothing at runtime. One repo, one deploy, one design system, one domain.

  **Keep it that way.** The decisive reason isn't convenience, it's correctness: **the pricing page is generated from `lib/plans`, the same module that drives real Stripe billing.** The marketing site therefore *cannot* advertise a price we don't charge. Split into a separate site and that becomes a copy-paste that silently drifts the day a plan changes ÔÇö the exact bug that ends with a customer holding a screenshot of a price we don't honour. One codebase makes that category of bug structurally impossible.

  The usual arguments for splitting don't hold here: marketing can't slow the app down (those pages never hit a server); an app outage can't take marketing down (static, CDN-cached); Vercel deploys are atomic and ~2 minutes; and for **SEO a single domain accruing authority is a reason to stay together, not to split**. The one real trigger to revisit is **non-engineers needing to edit copy without a PR** ÔÇö that's a Webflow/Framer decision, and we're not there.

  **The question worth asking is about DOMAINS, not codebases.** One host currently serves three different products:

  | Surface | Today | Conventional |
  |---|---|---|
  | Marketing | `/` | `bounce.com` (apex) |
  | Operator app | `/dashboard`, ÔÇŽ | `app.bounce.com` |
  | Storefronts | `/s/[slug]` | the operator's own domain |

  The strongest argument for moving the operator app to an `app.` subdomain is **cookie scoping**: the Supabase auth cookie is set on the apex, so it's transmitted on *every* request to the marketing pages **and to every public storefront** ÔÇö surfaces that have no business seeing an operator's session token. A subdomain confines it to the app. It also separates marketing-funnel analytics from product usage, and frees the apex to move to a CMS later without touching app code.

  **Plan:** when a real domain is bought, do the split at the **DNS layer** ÔÇö apex Ôćĺ marketing, `app.` Ôćĺ operator app ÔÇö which is Vercel config plus a middleware tweak, not a rewrite. Not urgent. The **storefront** custom-domain work is the more valuable half anyway (operators will want `book.bouncehouseusa.com`, not `bounce.com/s/bounce-usa`) ÔÇö tracked under **Multi-tenancy (Tier 0)** and the embeddable-widget follow-ups (Tier 3).
