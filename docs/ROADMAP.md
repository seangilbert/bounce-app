# Bounce — Production MVP Roadmap

Path from the current working demo to a production-ready, **multi-tenant** SaaS that can take real customers and pay real operators.

_Last updated: 2026-07-05._

## Where we are

The core product loop works end-to-end and is deployed (https://bounce-app.vercel.app):
conversational AI quote → deposit/full checkout (Stripe) → e-signed booking (SignWell)
→ operator dashboard / calendar / inbox, all on live data.

**Solid today**
- Booking + availability engine (date ranges, peak-reservation math, oversell detection, abandoned-checkout expiry).
- Stripe Checkout with idempotent webhooks; SignWell agreements auto-sent.
- Conversational AI quote agent grounded in live inventory + DB prices.
- Operator app + storefront, wired to real data, responsive, deployed.

**Known gaps (verified in code)**
- **No authentication** — middleware only refreshes Supabase cookies; operator app is open to anyone. `getDefaultOperator()` (16 call sites) hardcodes one operator.
- **Single Stripe account, no Connect** — all payments land in one account.
- **No transactional email** — no receipts/confirmations/operator alerts.
- **In-memory rate limiting** — per-instance, ineffective across serverless.
- **No automated tests / CI; hand-applied migrations; shared dev↔prod DB.**

Follow-ups: full focus-trap tab-cycling; preload adjacent slides._
- [~] **Settings** — `/settings` (was stub): editable **business profile** (name, owner, phone, geocoded service area, **timezone** — drives "today" for calendar/route/dashboard, migration `0019`), **booking policies** now operator-configurable (deposit %, auto-quote cap, min lead time — wired into checkout, storefront, and the AI escalation gate), and a **storefront/account** panel (storefront URL, plan/subscription, Connect status + connect). - [x] **Booking management** — `/bookings/[id]` detail page (operator-scoped) with actions: mark delivered / completed, cancel (frees inventory), and **refund** (Stripe refund → order refunded + booking canceled). Wired from the calendar's "Open booking". _Follow-up: edit a booking's items/dates; operator-created "New booking"._
- [x] **Driver daily route interface (deliveries)** — `/deliveries` is now a **mobile-first daily route** (operator-only, behind the existing login). Drop-offs (bookings **starting** the day) + pick-ups (bookings **ending** the day) from committed bookings (`paid`→`completed`), ordered by delivery window; single-day rentals appear in both. Each stop shows a type badge, time window, customer, items, and address with one-tap **Open in Maps**, **Call/Text** (customer phone now captured at checkout → `customer_phone`, migration `0018`) or an Email fallback, a Booking link, and **Mark delivered / picked up** (reuses `markDeliveredAction` / `markCompletedAction` → advances status, revalidates calendar + dashboard + deliveries). Day switcher (prev / next / Today via `?d=`). Route data verified against live bookings. _Setup: run migration `0018` (adds `customer_phone`) — **required before deploy**, since booking-create + the route query both reference it. Follow-ups: **driver access** (shareable per-day link or a driver role); proof-of-delivery (photo/signature) + per-stop notes; geographic route optimization, map view, multi-driver assignment._
- [x] **Wire the Inquiries inbox actions** — "Send reply" records the operator's (edited) reply + marks the inquiry `replied` (clears it from "needs you"); "Dismiss" → `dismissed` (hidden). Operator-scoped server actions; dashboard/badge counts update. Customer delivery via email (Resend) is wired. _Follow-up: persistent conversation, see next item._
- [~] **Inquiries → persistent conversation thread** — **Phase 1 (outbound) done.** An `inquiry_messages` table (migration `0021`) is now the conversation thread — senders `customer` / `operator` / `ai`, backfilled from the existing `inbound_message`, auto `ai_summary`, and `operator_reply`. The inbox detail renders the **full chronological thread** and the **composer is always available on every inquiry regardless of status** — each send appends to the thread, emails the customer (Resend), and updates status; the AI draft is now a *suggestion* that pre-fills the composer for `needs_review` (no longer the only way to reply). This kills the one-shot dead-end. `createInquiry` seeds the thread (customer message + AI auto-answer), `replyToInquiry` appends an operator message (and still sets `operator_reply` for the list preview). _Setup: run migration `0021` — **required before deploy** (inbox load, inquiry creation, and replies all use the table)._ **Phase 2 (inbound two-way) remains — its own item below.**
- [ ] **Inquiries Phase 2 — inbound two-way messaging** — the thread is outbound-only today (operator → customer via email); customers can't reply *back into* the app, so their responses land in the operator's own email inbox instead of the thread. Capture inbound replies into the same `inquiry_messages` thread as `sender = 'customer'`. Needs: (1) **per-inquiry routing** so a reply maps to the right thread — a plus-addressed reply-to on outbound email (`inbox+<inquiryId>@<domain>`) and/or a per-operator phone number for SMS; (2) an **inbound webhook** — Resend inbound email (verify + parse sender/body, strip quoted reply chains, match the inquiry, append a `customer` message) and/or **Twilio SMS**; (3) **inbox surfacing** — a new inbound reply reopens the inquiry (back to `needs_review`/unread), updates the badge counts, and fires the operator "new reply" notification. Considerations: verify the sender matches the inquiry's customer, spam/auto-reply-loop guards, attachments. Bigger lift — new external setup (verified inbound domain and/or provisioned phone number) plus the webhook endpoint + matching logic.
- [x] **Inquiry loop — contact capture + honest delivery** — closed the outbound gap. The storefront chat never collected an email, so an escalated ("needs you") inquiry had no reply-to and hitting **Send** in the inbox silently delivered nothing (the seed inquiries have emails, which masked it). Now: when the AI **escalates** (status `review`), the chat shows an inline **"leave your email and we'll send your custom quote"** capture → `POST /api/inquiries/contact` attaches it to the inquiry (operator-scoped, rate-limited). And the inbox composer is **honest about delivery** — "Emails {address}" when contact is on file, or a **"No email — can't deliver"** warning with **Send disabled** when it isn't. Verified the capture endpoint + operator scoping; the assistant already emits `review` on escalation. _(Customers replying back into the thread is still Inquiries Phase 2.)_
- [x] **Inquiry → booking outcome indicator** — the inbox now shows whether each inquiry **converted** (especially useful on auto-answered ones, which the operator never saw): a **Booked / Checkout-started / Canceled / No-booking-yet** banner at the top of the detail (with amount, event date, and a link to the booking) + a compact badge on the list card. Linked two ways: **explicit** — booking from the chat quote writes `inquiry.booking_id` (threaded chat → checkout → `/api/bookings` → `linkInquiryToBooking`); **heuristic** — otherwise matched by customer email + overlapping dates (batched into two booking queries, not N+1). Pairs with the persistent thread: spot a non-converted auto-response and follow up. Both paths verified; no migration (the `inquiries.booking_id` column already existed).
- [x] **Balance collection** — booking detail shows the balance due with **Collect balance** (Stripe checkout for total − deposit; webhook marks it paid-in-full without re-confirming/re-sending) and **Mark paid (cash)**. Charges only the remaining amount; doesn't reset the committed booking.

## Tier 2 — Reliability & scale

- [ ] **Performance (infra) follow-ups** — _(app-level nav perf: cached session/operator resolution, parallelized queries, loading skeletons; **now also**: the **redundant per-nav RSC `auth.getUser()` is eliminated** — the middleware forwards the verified user id via a trusted, inbound-stripped header (`x-operator-user-id`) and Server Components read that (falling back to `getUser` when absent); **session→operator is a single join query** instead of two round-trips; and the middleware **skips `getUser` entirely on public routes** (storefront/APIs). ~430ms less session overhead per operator navigation.)_ Remaining (infra, needs setup): (a) **co-locate Supabase + Vercel in the same region** — each DB query is ~130ms; cross-region latency adds up. (b) **local JWT verification** (`getClaims` w/ asymmetric signing keys) to also drop the ~300ms `auth.getUser()` the **middleware** still makes on every operator request — requires enabling asymmetric JWT keys in the Supabase dashboard first.
- [x] **Overbooking race hardening** — the reserve step (booking → `pending_payment` at checkout) is now atomic. A Postgres `reserve_booking(booking_id)` function (migration `0020`) locks the booking's item rows (`FOR UPDATE`, ordered by id to avoid deadlocks), re-checks **peak** capacity over the rental range excluding this booking, then flips the status — so two simultaneous checkouts for the last unit serialize and the loser raises `OVERSELL`. `/api/checkout` calls it **before** creating the Stripe session and returns a clean 409 "just sold out" (surfaced in the storefront checkout). _Setup: run migration `0020` — **required before deploy** (checkout calls the RPC). Verified with a concurrent-reservation test. Note: the post-paid oversell guard in `markBookingPaid` stays as a logging backstop._
- [ ] **Error monitoring** (Sentry) + structured logging.
- [ ] **Automated tests** — none exist; prioritize the availability engine, quote/escalation logic, webhook handlers. Add CI (typecheck/build/test gates) + a staging environment.
- [ ] **Accessibility pass** + mobile QA.
