# Bounce — Production MVP Roadmap

Path from the current working demo to a production-ready, **multi-tenant** SaaS that can take real customers and pay real operators.

_Last updated: 2026-07-16._

## Where we are

The core product loop works end-to-end and is deployed (https://bounce-app.vercel.app):
conversational AI quote → deposit/full checkout (Stripe) → e-signed booking (SignWell)
→ operator dashboard / calendar / inbox, all on live data.

**Solid today**
- Booking + availability engine (date ranges, peak-reservation math, oversell detection, abandoned-checkout expiry).
- Stripe Checkout with idempotent webhooks; SignWell agreement send path built + verified (currently **off in prod** — test mode; see Go-Live Checklist).
- Conversational AI quote agent grounded in live inventory + DB prices.
- Operator app + storefront, wired to real data, responsive, deployed.

**Known gaps (verified in code)**
- **No RLS policies** — RLS is *enabled* on the tenant tables but the only policy in the whole migration set is `operator_members` "members read own" (`0008`). Every read/write goes through `createAdminClient()` (service role, bypasses RLS), so **tenant isolation lives entirely in application code**: one missed `.eq("operator_id", …)` leaks another operator's data with no second line of defense. Tracked under Multi-tenancy (Tier 0).
- **Migrations are hand-applied** — CLI-tracked (`./scripts/db-migrate.sh`), but not part of the deploy.
- ~~No authentication~~ ✅ Supabase Auth + `operator_members` + middleware gating. ~~Single Stripe account, no Connect~~ ✅ Express connected accounts + destination charges. ~~No transactional email~~ ✅ Resend. ~~In-memory rate limiting~~ ✅ Upstash Redis (sliding window). ~~No automated tests / CI~~ ✅ **Vitest (221 tests) + GitHub Actions CI**. ~~Shared dev↔prod DB~~ ✅ split into separate projects.

### RLS — ✅ resolved (was "the one code item that should worry you")

**This was the top code-level risk; it's now closed (2026-07-18).** Tenant isolation used to live *entirely* in hand-written application code — every query on the service-role client, so one forgotten `.eq("operator_id", …)` would leak another operator's data with no second line of defense. The pattern had already bitten twice (both harmlessly, both by luck): the middleware gate missing `/customers`/`/promos`/`/documents`/`/account` (fixed `5afc478` + a route-group test), and every renter-portal query being hand-scoped with correctness proven by tests rather than by the DB refusing wrong rows.

Now the database enforces it (belt **and** braces), across migrations `0049`–`0057` + a lock test — see the full **"Enable RLS as tenant-isolation defense-in-depth"** item below the Tier list for the phase-by-phase detail:
- **Baseline (`0050`)** — RLS enabled on every table; this **found + fixed a live PII leak** (the public anon key could read `customer_accounts` — every customer's email/name — via PostgREST).
- **Renter portal (`0049`) + every convertible operator surface (`0051`–`0057`)** — moved onto the user-scoped client behind membership-keyed policies, so a forgotten filter (or a JWT hitting PostgREST directly) can't cross tenants. Verified per-surface with minted JWTs (own-row works AND cross-operator blocked at the DB) + anon-blocked on prod.
- **Anon/public reads (Phase 3)** — deliberately **kept on the service-role client** with explicit column selection: the public catalog has no tenant boundary, and a row-level anon policy on `operators` would expose Stripe columns. Correct pattern, not a gap.
- **Audit + lock (`service-role-lock.test.ts`)** — the service-role client is confined to a 34-file audited allowlist; a new user-facing importer fails CI. Locks the whole thing against regression.

- **DECISION — keep the marketing site inside the app; split at DNS, not at the codebase** _(2026-07-14)_

  **Where it lives today:** a `(marketing)` route group in the same Next.js app, **statically prerendered** (`○` in the build output) and served from Vercel's CDN. The middleware skips those paths entirely, so they cost nothing at runtime. One repo, one deploy, one design system, one domain.

  **Keep it that way.** The decisive reason isn't convenience, it's correctness: **the pricing page is generated from `lib/plans`, the same module that drives real Stripe billing.** The marketing site therefore *cannot* advertise a price we don't charge. Split into a separate site and that becomes a copy-paste that silently drifts the day a plan changes — the exact bug that ends with a customer holding a screenshot of a price we don't honour. One codebase makes that category of bug structurally impossible.

  The usual arguments for splitting don't hold here: marketing can't slow the app down (those pages never hit a server); an app outage can't take marketing down (static, CDN-cached); Vercel deploys are atomic and ~2 minutes; and for **SEO a single domain accruing authority is a reason to stay together, not to split**. The one real trigger to revisit is **non-engineers needing to edit copy without a PR** — that's a Webflow/Framer decision, and we're not there.

  **The question worth asking is about DOMAINS, not codebases.** One host currently serves three different products:

  | Surface | Today | Conventional |
  |---|---|---|
  | Marketing | `/` | `bounce.com` (apex) |
  | Operator app | `/dashboard`, … | `app.bounce.com` |
  | Storefronts | `/s/[slug]` | the operator's own domain |

A subdomain confines it to the app. It also separates marketing-funnel analytics from product usage, and frees the apex to move to a CMS later without touching app code.
